# Архитектура StreakBot

Документ описывает ключевые архитектурные решения и причины, по которым они
приняты.

## Поток данных

```
Telegram Update
      │
ActivityMiddleware (фиксирует время последней активности)
      │
DatabaseMiddleware (открывает сессию, внедряет repo)
      │
RegistrationMiddleware (проверка регистрации / онбординга)
      │
Router → Handler
      │
Service (бизнес-логика: streak, geo, scheduler)
      │
Repository (CRUD)
      │
SQLAlchemy (async session)
      │
SQLite / PostgreSQL
```

## Слои и ответственность

| Слой | Где | Отвечает за |
|---|---|---|
| Конфиг | `bot/config.py` | Загрузка и валидация `.env` (pydantic-settings) |
| Тексты | `bot/constants.py` | Все строки сообщений и структурные константы |
| Хендлеры | `bot/handlers/*` | Маршрутизация апдейтов, FSM-сценарии |
| Клавиатуры | `bot/keyboards/builders.py` | Все reply/inline-клавиатуры |
| Сервисы | `bot/services/*` | Стрики, геокодинг, планировщик |
| Репозиторий | `bot/database/repository.py` | Единственная точка доступа к БД |
| Модели | `bot/database/models.py` | ORM-схема (User, Task, TaskLog) |
| Middlewares | `bot/middlewares/*` | DI сессии, гейт регистрации, учёт активности |
| Утилиты | `bot/utils/validators.py` | Парсинг времени/UTC, экранирование MarkdownV2 |

## Ключевые решения

### Стрик вычисляется, а не хранится
Стрик нигде не лежит как поле. Он считается на лету в `services/streak.py` по
записям `TaskLog`. Это исключает рассинхронизацию: единственный источник
правды — журнал отметок. Одноразовые задачи (`one_time`) в стриках не
участвуют: в карточке у них нет строк стрика, и в `/stats` они не показываются.

### Отметка выполнения — только из дайджестов и напоминаний
Отдельных команд `/today` / `/done` нет. Пользователь отмечает выполнение
интерактивно: из утреннего и вечернего дайджестов (см. «Планировщик») и кнопкой
«Выполнена» на напоминании. Это единственные точки входа в отметку.

### Добавление задачи: пошаговый FSM с «‹ Назад»
`/add` ведёт по шагам: название → частота → (дни недели / дата) → напоминание →
(время). Inline-нажатия редактируют одно сообщение, текстовый ввод (название,
время) порождает новое. На каждом шаге **после ввода названия** (кроме первого —
выбора частоты) есть кнопка «‹ Назад» отдельной строкой внизу: она редактирует
сообщение и возвращает к предыдущему шагу, сохраняя прежний выбор (например,
отмеченные дни недели). Цель возврата зависит от ветки частоты
(`daily` / `specific_days` / `one_time`) и текущего FSM-состояния; callback-data
шагов назад — `add_back_*`.

### Редактирование задачи (/edit)
`/edit` показывает список задач (клавиатура идентична `/delete`, но с callback-
data `edit_select` / `edit_page`). Выбор задачи открывает карточку (название,
частота, напоминание) с пунктами «Название», «Частота», «Напоминание» и «‹ Назад»
(к списку) — всё редактированием одного сообщения.

- **Название** и **время напоминания** вводятся текстом: бот присылает НОВОЕ
  сообщение с запросом, после успешного ввода — сообщение с кнопкой
  «‹ Вернуться к задаче» (`edit_to_card`), которая редактируется обратно в
  карточку. Некорректный ввод — повтор с сообщением об ошибке; дубликат имени
  (без учёта регистра, исключая саму задачу) запрещён.
- **Частота** меняется редактированием карточки: «Каждый день» применяется сразу,
  «В конкретные дни» открывает выбор дней (та же клавиатура, что в `/add`) →
  «Готово». Доступны только `daily` и `specific_days`; `one_time_date`
  сбрасывается.
- **Напоминание**: если его нет — запрос времени новым сообщением; если есть —
  меню «Изменить» / «Убрать» / «‹ Назад» (редактированием карточки). При смене
  частоты или времени job напоминания пересоздаётся (`replace_existing`), при
  удалении — снимается.
- Везде, где у экрана есть предыдущее состояние в рамках одного сообщения, есть
  «‹ Назад» (`edit_to_card` — к карточке, `edit_freq_back` — к выбору частоты).
  Стрик и логи задачи при редактировании не трогаются.
- `edit_to_card` ограничен состояниями `EditTaskStates` (`StateFilter`). Кнопка
  «‹ Вернуться к задаче» живёт на отдельном сообщении об успехе и может остаться
  «устаревшей»: без фильтра нажатие на неё, пока пользователь уже в другом
  сценарии (например, в /delete), сбрасывало бы чужой FSM и ломало команду.

### Просмотр и управление задачами (/tasks)
`/tasks` (`handlers/tasks.py`) — навигация по задачам в рамках одного сообщения.
Иерархия: **меню → список раздела → карточка задачи → подтверждение удаления**.

**Контекст в callback-data, без FSM.** Раздел (`a` — все, `t` — на сегодня) и
страница (а для карточки ещё и `task_id`) кодируются прямо в callback-data
(`tk_list:{sec}:{page}`, `tk_card/tk_done/tk_del/tk_dok:{sec}:{page}:{task_id}`).
Поэтому навигация не зависит от FSM (переживает сброс командой), а «‹ Назад» на
каждом шаге детерминированно ведёт к предыдущему состоянию (карточка → список той
же страницы → меню; подтверждение → карточка).

- **Меню**: «Все задачи» (`tk_list:a:0`) и «Задачи на сегодня» (`tk_list:t:0`).
- **Список** (`tk_list`): жирный заголовок, по каждой задаче — название и стрик
  (для одноразовых стрика нет), жирная подсказка. Клавиатура: кнопки задач по 2 в
  ряд (последняя нечётная — одна) + стрелки «‹»/«›» (alert на краях) + «‹ Назад» к
  меню. «Все задачи» — `stats_tasks` (активные без одноразовых, как /stats);
  «Задачи на сегодня» — `get_tasks_due_on(today)` (любого типа). По 5 на странице.
- **Карточка** (`tk_card`): название, параметры (частота, напоминание), стрик,
  подсказка. Кнопка-галочка «Выполнено» (☑️/⬜) показывается только если задача
  запланирована на сегодня; `tk_done` переключает статус сегодняшнего лога сразу в
  БД и перерисовывает карточку. Удаление (`tk_del`) запрашивает подтверждение;
  `tk_dok` мягко удаляет задачу и снимает напоминание (`scheduler`), затем
  возвращает к списку раздела (страница клампится — задач стало меньше).
- Связность без циклов: `tasks.py` переиспользует `stats_tasks` из `stats.py` и
  `get_*_streak` из `services/streak.py`; маленькие view-хелперы (частота, дата по
  таймзоне) продублированы локально, чтобы не тянуть приватные функции хендлеров.

### Отметка задач списком (/check)
`/check` (`handlers/today.py`) присылает компактный список задач на сегодня:
жирный заголовок «Задачи на сегодня», блок невыполненных (обычный текст), блок
выполненных (зачёркнутый, `~…~`) и жирную подсказку, плюс кнопку «Отметить»
(`check_open`). Логи на сегодня при необходимости создаются (как при сборке
дайджеста).

- **«Отметить»** меняет ТОЛЬКО клавиатуру на галочки (`task_select_kb(...,
  done_text="Сохранить", with_back=False)` — тот же билдер, что у дайджестов, но
  кнопка завершения «Сохранить» и без «‹ Назад»). Стартовые галочки = уже
  выполненные задачи.
- **Смена галочки** (`check_toggle`) редактирует и текст, и клавиатуру: разбиение
  на блоки идёт по текущему выбору из FSM (`check_selected`), поэтому отмеченная
  задача в реальном времени уходит в нижний (зачёркнутый) блок, снятая —
  возвращается в верхний. БД при этом не меняется.
- **«Сохранить»** (`check_done`) фиксирует статусы (`_apply_today_marks`:
  выбранные → `done`, остальные → `pending`) и возвращает список с актуальными
  статусами и кнопкой «Отметить» (подсказка меняется на «Нажми на кнопку…»).
- Робастность как у `tm_*`/`md_*`: вход (`check_open`) — без фильтра состояния
  (работает и после рестарта), шаги — под `CheckStates.marking` (отдельным от
  `MarkingStates`, чтобы флоу /check и дайджестов не перехватывали кнопки друг
  друга). Любая команда сбрасывает FSM → «устаревшие» галочки/«Сохранить» просто
  не срабатывают.

### Repository pattern
Хендлеры и сервисы не пишут SQL. Вся работа с БД — через методы `Repository`,
привязанного к одной async-сессии. Сессию на каждый апдейт открывает
`DatabaseMiddleware` и коммитит её по завершении обработки (откат при ошибке).

### Dependency Injection через aiogram
`session` и `repo` пробрасываются в хендлеры через `data` (middleware).
`config` и `scheduler` — через `dp.workflow_data`. Хендлеры ничего не создают
сами и принимают зависимости как аргументы.

### MarkdownV2 и экранирование
Тексты в `constants.py` хранятся в «чистом» виде. При отправке любая строка
проходит через `escape_md`, поэтому пользовательский ввод (названия задач,
города) не ломает разметку. Жирный заголовок шагов добавляется программно
вокруг уже экранированного текста.

### Порядок роутеров
`cancel → onboarding → start → add_task → today → delete_task → edit_task →
stats → tasks → settings`.

- `cancel` первым — перехватывает `/cancel` в любом состоянии: очищает
  FSM-состояние **и** убирает reply-клавиатуру (полное обнуление, как будто
  активного действия не было).
- `onboarding` идёт **перед** `start`. Во время регистрации любая команда,
  кроме `/start`, должна трактоваться как некорректный ввод и повторять текущий
  шаг. Обработчики онбординга перехватывают всё, кроме `/start` (через
  `~CommandStart()`), поэтому `/help`, `/stats` и прочие команды во время
  онбординга не «проваливаются» в свои хендлеры. После регистрации онбординг-
  состояний нет, и команды доходят до своих роутеров, сбрасывая активное
  состояние (FSM-шаги основного режима исключают команды через
  `~Command(*COMMANDS)`).

### Восстановление после рестарта
FSM хранится в памяти (`MemoryStorage`) и теряется при перезапуске.
- Незавершённый онбординг: `RegistrationMiddleware` при отсутствии состояния
  перезапускает регистрацию с приветствия.
- Scheduler-jobs: при старте `SchedulerService.restore_jobs` заново создаёт
  утренние/вечерние дайджесты и напоминания для всех зарегистрированных
  пользователей и их активных задач по данным из БД.

### Планировщик
`APScheduler AsyncIOScheduler`. Job ID детерминированы
(`morning_{uid}`, `evening_{uid}`, `reminder_{task_id}`), что позволяет
пересоздавать их идемпотентно (`replace_existing=True`).

**Откладывание при активности.** Перед отправкой дайджеста сверяется время
последней активности (in-memory словарь, обновляется `ActivityMiddleware`).
Если разница меньше 5 минут — отправка откладывается ровно на остаток до 5 минут
(одноразовый job `*_retry_{uid}`), и при следующей попытке разница считается
заново (отложки не суммируются). Максимум 3 отложки — на 4-й итерации
уведомление уходит безусловно.

**Не дважды за день.** Факт отправки фиксируется в словаре `digest_sent`
(`(uid, 'morning'|'evening') -> дата`). Если дайджест за сегодня уже отправлен,
смена времени уведомления или часового пояса в настройках его не переотправит —
новые настройки вступают в силу со следующего дня.

**Чистое состояние при отправке.** Непосредственно перед отправкой сбрасывается
FSM-состояние пользователя (через `storage`) и убирается reply-клавиатура.
Исключение: если у дайджеста есть inline-кнопка отметки задач, reply-клавиатуру
в этом сообщении снять нельзя (одно сообщение не несёт и inline-, и reply-
разметку); FSM-состояние сбрасывается всегда.

- **Утренний дайджест** (`build_morning_digest`): блок задач на сегодня + блок
  «Просроченные задачи» (вчерашние невыполненные задачи **любого типа** — у
  которых есть вчерашний лог не со статусом `done`). Клавиатура: сверху кнопка
  отметки сегодняшних задач (`tm_*`, если на сегодня есть задачи), ниже — кнопка
  отметки вчерашних просроченных (`md_*`, если просроченные есть).
- **Вечерний дайджест** (`build_evening_digest`): итоги дня (выполнено / осталось)
  + кнопка отметки сегодняшних задач (`tm_*`, если на сегодня есть задачи).
  Удалённые (`is_active=False`) задачи в итоге не показываются, даже если за
  сегодня по ним остался лог.
- **Отметка сегодняшних задач — общий флоу `tm_*`** (утренний и вечерний
  дайджест; хендлеры в `handlers/today.py`). Кнопка (`tm_mark:{origin}`, origin =
  `morning`/`evening`) редактирует **то же сообщение** в экран выбора. Экран
  показывает ВСЕ сегодняшние задачи: невыполненные без галочки, выполненные — с
  галочкой; галочки можно как ставить, так и снимать (сетка 2×N, пагинация при >6).
  «Готово» → экран подтверждения нового статуса всех задач → «Подтвердить»
  фиксирует: выбранные → `done`, остальные → `pending`, и возвращает к дайджесту.
  Кнопка после подтверждения остаётся; её текст — «Отменить выполнение», если все
  задачи выполнены, иначе «Отметить выполненные». Источник флоу (`tm_origin` в FSM)
  определяет, к какому дайджесту вернуться; выбор — в `tm_selected`.
- **Отметка вчерашних просроченных — флоу `md_*`** (только утро): экран выбора с
  галочками → «Готово» → подтверждение → «Подтвердить». Отмеченные вчерашние логи
  → `done`, неотмеченные → `missed`; есть предупреждение о сгорании; «Подтвердить»
  активно до 12:00 по времени пользователя, после — экран «время вышло» и
  финальный вид.
- **Защита от «устаревших» кнопок (`MarkingStates.active`).** Кнопка входа
  (`tm_mark`/`md_mark`) остаётся без фильтра состояния — чтобы работать и после
  рестарта, когда FSM пуст, — но выставляет состояние `MarkingStates.active`;
  все последующие шаги (`tm_*`/`md_*`) требуют его. Любая команда вызывает
  `state.clear()`; если пользователь между шагами ушёл в другую команду
  (`/edit`, `/delete`, `/tasks` …), нажатия на старом экране отметки просто не
  срабатывают (а не приводят к подстановке не того дайджеста или к
  рассинхронизации текста и кнопок). `tm_back_digest`/`tm_confirm`/`md_back_digest`/
  `md_confirm`/`md_expired_ok` по завершении очищают состояние; промежуточные шаги
  выбора/подтверждения его сохраняют. Это также гарантирует, что при срабатывании
  шага `tm_origin`/выбор ещё на месте.
- **Напоминание о задаче** (`send_reminder`): сообщение несёт inline-кнопку
  «Выполнена» (`rem_done:{task_id}:{дата}`). Нажатие отмечает задачу выполненной
  за указанную дату; если она уже отмечена — сообщает об этом. Кнопка активна
  до 12:00 следующего дня, после — «время вышло».
- **Глобальный job `check_and_expire_tasks`** (00:05 UTC): переводит
  просроченные `pending`-логи в `missed` и «забывает» одноразовые задачи —
  через сутки после истечения срока (`one_time_date < today − 1`) задача
  деактивируется, а её напоминание снимается.

### Часовые пояса
Время дайджестов и напоминаний привязано к таймзоне пользователя. Пояс
определяется по пресету, вводу `UTC±N` (валидатор `parse_utc_offset` →
`Etc/GMT∓N`) или названию города (`services/geo.py`: geopy + timezonefinder).
Все внешние вызовы геокодера обёрнуты в try/except.

## Обработка ошибок
- Внешние вызовы (геокодер, Telegram API) обёрнуты в try/except.
- Блокировка бота пользователем (`TelegramForbiddenError`) → `is_active=False`.
- Глобальный `dp.errors` логирует исключение и показывает пользователю
  нейтральное сообщение — необработанные трейсбеки наружу не утекают.
