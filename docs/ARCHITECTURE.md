# Архитектура StreakBot

Документ описывает ключевые архитектурные решения и причины, по которым они
приняты.

## Поток данных

```
Telegram Update
      │
DatabaseMiddleware (открывает сессию, внедряет repo)
      │
RegistrationMiddleware (проверка регистрации / онбординга)
      │
Router → Handler
      │
Service (бизнес-логика: streak, geo, scheduler)
      │
Repository (CRUD)
      │
SQLAlchemy (async session)
      │
SQLite / PostgreSQL
```

## Слои и ответственность

| Слой | Где | Отвечает за |
|---|---|---|
| Конфиг | `bot/config.py` | Загрузка и валидация `.env` (pydantic-settings) |
| Тексты | `bot/constants.py` | Все строки сообщений и структурные константы |
| Хендлеры | `bot/handlers/*` | Маршрутизация апдейтов, FSM-сценарии |
| Клавиатуры | `bot/keyboards/builders.py` | Все reply/inline-клавиатуры |
| Сервисы | `bot/services/*` | Стрики, геокодинг, планировщик |
| Репозиторий | `bot/database/repository.py` | Единственная точка доступа к БД |
| Модели | `bot/database/models.py` | ORM-схема (User, Task, TaskLog) |
| Middlewares | `bot/middlewares/*` | DI сессии, гейт регистрации |
| Утилиты | `bot/utils/validators.py` | Парсинг времени/UTC, экранирование MarkdownV2 |

## Ключевые решения

### Стрик вычисляется, а не хранится
Стрик нигде не лежит как поле. Он считается на лету в `services/streak.py` по
записям `TaskLog`. Это исключает рассинхронизацию: единственный источник
правды — журнал отметок.

### Repository pattern
Хендлеры и сервисы не пишут SQL. Вся работа с БД — через методы `Repository`,
привязанного к одной async-сессии. Сессию на каждый апдейт открывает
`DatabaseMiddleware` и коммитит её по завершении обработки (откат при ошибке).

### Dependency Injection через aiogram
`session` и `repo` пробрасываются в хендлеры через `data` (middleware).
`config` и `scheduler` — через `dp.workflow_data`. Хендлеры ничего не создают
сами и принимают зависимости как аргументы.

### MarkdownV2 и экранирование
Тексты в `constants.py` хранятся в «чистом» виде. При отправке любая строка
проходит через `escape_md`, поэтому пользовательский ввод (названия задач,
города) не ломает разметку. Жирный заголовок шагов добавляется программно
вокруг уже экранированного текста.

### Порядок роутеров
`cancel → onboarding → start → add_task → today → delete_task → stats → settings`.

- `cancel` первым — перехватывает `/cancel` в любом состоянии.
- `onboarding` идёт **перед** `start`. Во время регистрации любая команда,
  кроме `/start`, должна трактоваться как некорректный ввод и повторять текущий
  шаг. Обработчики онбординга перехватывают всё, кроме `/start` (через
  `~CommandStart()`), поэтому `/help`, `/today` и прочие команды во время
  онбординга не «проваливаются» в свои хендлеры. После регистрации онбординг-
  состояний нет, и команды доходят до своих роутеров, сбрасывая активное
  состояние (FSM-шаги основного режима исключают команды через
  `~Command(*COMMANDS)`).

### Восстановление после рестарта
FSM хранится в памяти (`MemoryStorage`) и теряется при перезапуске.
- Незавершённый онбординг: `RegistrationMiddleware` при отсутствии состояния
  перезапускает регистрацию с приветствия.
- Scheduler-jobs: при старте `SchedulerService.restore_jobs` заново создаёт
  утренние/вечерние дайджесты и напоминания для всех зарегистрированных
  пользователей и их активных задач по данным из БД.

### Планировщик
`APScheduler AsyncIOScheduler`. Job ID детерминированы
(`morning_{uid}`, `evening_{uid}`, `reminder_{task_id}`), что позволяет
пересоздавать их идемпотентно (`replace_existing=True`). Глобальный job
`check_and_expire_tasks` в 00:05 UTC переводит просроченные `pending`-логи в
`missed`.

### Часовые пояса
Время дайджестов и напоминаний привязано к таймзоне пользователя. Пояс
определяется по пресету, вводу `UTC±N` (валидатор `parse_utc_offset` →
`Etc/GMT∓N`) или названию города (`services/geo.py`: geopy + timezonefinder).
Все внешние вызовы геокодера обёрнуты в try/except.

## Обработка ошибок
- Внешние вызовы (геокодер, Telegram API) обёрнуты в try/except.
- Блокировка бота пользователем (`TelegramForbiddenError`) → `is_active=False`.
- Глобальный `dp.errors` логирует исключение и показывает пользователю
  нейтральное сообщение — необработанные трейсбеки наружу не утекают.
